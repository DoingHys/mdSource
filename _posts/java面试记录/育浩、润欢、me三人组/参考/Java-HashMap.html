HashMap是一个散列表，它存储的内容是键值对key-value映射。先看看HashMap的大致结构，然后从源码开始。

1. 类的定义

public class HashMap<K,V> extends AbstractMap<K,V>
implements Map<K,V>, Cloneable, Serializable

HashMap继承AbstractMap抽象类，AbstractMap中定义了存储key的Set<K> keySet;，Set是不会出现重复值的，所以HashMap中key不存在重复值的情况，还定义了存储value的Collection<V> values;，Collection是允许存在重复值的。AbstractMap实现Map接口，实现了put、remove、get等方法，方法的具体实现将在下面类的方法中探究。

这里发现一个问题，为什么AbstractMap实现Map接口，而HashMap也实现了Map接口。在Stackoverflow上有一个答案：[Why does LinkedHashSet extend HashSet and implement Set]。

  • 在HashMap中定义了一些静态变量，作为未指定时的默认值。如：
xxxxxxxxxx
static final int DEFAULT_INITIAL_CAPACITY = 1 << 4;

定义HashMap初始化容量，这里是用的是位运算，1 左移 4 位，0000 0001 -> 0001 0000 变成了 16。

xxxxxxxxxx
static final int MAXIMUM_CAPACITY = 1 << 30;

定义最大初始化容量，同样使用位运算，1 左移 30 位，变成 1 073 741 824。

xxxxxxxxxx

定义负载因子，就是当 已用容量/总容量 > 0.75 时，对HashMap进行扩容。

xxxxxxxxxx
static final int TREEIFY_THRESHOLD = 8;

在JDK 1.8 后HashMap默认某个链表的长度达到 8 后就会进化成红黑树。

xxxxxxxxxx
static final int UNTREEIFY_THRESHOLD = 6;

当红黑树的元素个数小于 6 的时候红黑树就会退化成链表。

xxxxxxxxxx
static final int MIN_TREEIFY_CAPACITY = 64;

这个值与TREEIFY_THRESHOLD值一起使用，当某个链表的长度大于TREEIFY_THRESHOLD时，还需要判断整个HashMap中元素的个数是否大于MIN_TREEIFY_CAPACITY的值。如果大于则进行转化红黑树，小于则不进行处理，这是为了避免在哈希表建立初期，多个键值对恰好被放入了同一个链表中而导致不必要的转化。

3.HashMap中的方法

  • hash方法是根据key值进行计算得到hash值。如果key为null，直接返回 0，插入到链表的第一位，哈希表的容量都是 2 的 N 次方，key的 hashCode() 在很多时候下低位是相同的，这将导致冲突（碰撞），因此 1.8 以后做了个移位操作：将元素的 hashCode() 和自己右移 16 位后的结果求异或，避免只靠低位数据来计算哈希时导致的冲突，计算结果由高低位结合决定，可以避免哈希值分布不均匀，异或运算能更好的保留各部分的特征。

    xxxxxxxxxx
    static final int hash(Object key) {
     int h;
     return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
    }

  • putval方法。通过Hash值与数组最大下标进行&操作，为什么这么做，首先得知道一条公式a = 2∧n 时，b % a = b & (a - 1)，而HashMap的数组长度总是 2的n次幂，旧主干数组的最大下标也是等于数组长度减 1，因为数组是从0开始的。所以这个&操作就相当于是求余操作，求余操作则是确保得出的数字一定在数组的范围内。确定数组位置后如果链表中已经有数据则插入到链表尾部。

    x
    final V putVal(int hash, K key, V value, boolean onlyIfAbsent,
                boolean evict) {
     Node<K,V>[] tab; Node<K,V> p; int n, i;
     //如果主干数组是空的，则进行一次扩容
     if ((tab = table) == null || (n = tab.length) == 0)
         n = (tab = resize()).length;
     //计算插入数组下标，与数组最大下标进行&操作，如果主干上当前位置没有元素，则直接插入新的元素
     if ((p = tab[i = (n - 1) & hash]) == null)
         tab[i] = newNode(hash, key, value, null);
     else {
         Node<K,V> e; K k;
         //如果主干上的hash值相同，并且key的值相同，则复制一份到e中。
         if (p.hash == hash &&
             ((k = p.key) == key || (key != null && key.equals(k))))
             e = p;
         //如果链表是红黑树，进行红黑树的put
         else if (p instanceof TreeNode)
             e = ((TreeNode<K,V>)p).putTreeVal(this, tab, hash, key, value);
         else {
             //开始遍历
             for (int binCount = 0; ; ++binCount) {
                 //如果遍历到最后，则直接插入元素
                 if ((e = p.next) == null) {
                     p.next = newNode(hash, key, value, null);
                     //如果当前节点数大于等于TREEIFY_THRESHOLD-1，则需要转换为红黑树
                     if (binCount >= TREEIFY_THRESHOLD - 1) // -1 for 1st
                         treeifyBin(tab, hash);
                     break;
                 }
                 //如果在链表上找到相同的key，直接退出循环。
                 if (e.hash == hash &&
                     ((k = e.key) == key || (key != null && key.equals(k))))
                     break;
                 p = e;
             }
         }
         //e不为空，则将e的value修改
         if (e != null) { // existing mapping for key
             V oldValue = e.value;
             if (!onlyIfAbsent || oldValue == null)
                 e.value = value;
             afterNodeAccess(e);
             return oldValue;
         }
     }
     //修改次数+1
     ++modCount;
     //查看当前元素数量是否大于负载因子*最大元素个数，超过则进行扩容
     if (++size > threshold)
         resize();
     afterNodeInsertion(evict);
     return null;
    }

  • resize方法。该方法对HashMap进行扩容，扩容时会新建一个主干数组，然后遍历旧数组将数据转移，最后达到扩容。值得一提的是在确定数据在新数组内的位置时，resize中用的是用数据的hash值与旧主干数组长度进行&操作，最后等于 0 的保持在原来的下标中，不于等 0 的将原数组的下标加上旧数组的长度就是在新数组中的长度。

    根据上文的putVal方法，为什么不是与新数组的最大下标进行&运算。首先看一下这几个计算。23%16=7; 23%32=23; 63%16=15； 63%32=31；10%16=10; 10%32=10;，通过几个计算，hash值大于原数组长度的，扩容后的余数等于原余数 + 数组长度（有点废话），而小于原数组长度的的余数还是它本身，在扩容的时候原链表中有些node是需要改变位置的，而有些node依然保留在原位置，其实就是把原链表分开成一条位置不变的链表loHead，一条需要移动位置的链表hiHead，然后分别放入到数组的位置。

    final Node<K,V>[] resize() {
         Node<K,V>[] oldTab = table;
         int oldCap = (oldTab == null) ? 0 : oldTab.length;
         int oldThr = threshold;
         int newCap, newThr = 0;
         // 对扩容进行条件判断
         if (oldCap > 0) {
             //查看原来容量是否已经是最大值
             if (oldCap >= MAXIMUM_CAPACITY) {
                 threshold = Integer.MAX_VALUE;
                 return oldTab;
             }
             //查看扩容后容量是否小于最大值，并且大于默认容量，则此时扩容为原来的2倍
             else if ((newCap = oldCap << 1) < MAXIMUM_CAPACITY &&
                      oldCap >= DEFAULT_INITIAL_CAPACITY)
                 newThr = oldThr << 1; // double threshold
         }
         else if (oldThr > 0) // initial capacity was placed in threshold
             newCap = oldThr;
         else {               // zero initial threshold signifies using defaults
             newCap = DEFAULT_INITIAL_CAPACITY;
             newThr = (int)(DEFAULT_LOAD_FACTOR * DEFAULT_INITIAL_CAPACITY);
         }
         if (newThr == 0) {
             float ft = (float)newCap * loadFactor;
             newThr = (newCap < MAXIMUM_CAPACITY && ft < (float)MAXIMUM_CAPACITY ?
                       (int)ft : Integer.MAX_VALUE);
         }
         threshold = newThr;
         @SuppressWarnings({"rawtypes","unchecked"})
             Node<K,V>[] newTab = (Node<K,V>[])new Node[newCap];
         table = newTab;
         if (oldTab != null) {
             //遍历旧的hash表
             for (int j = 0; j < oldCap; ++j) {
                 Node<K,V> e;
                 if ((e = oldTab[j]) != null) {
                     oldTab[j] = null;
                     //如果当前链表中就一个元素，重新计算位置
                     if (e.next == null)
                         newTab[e.hash & (newCap - 1)] = e;
                     //如果是红黑树，使用红黑树的方法
                     else if (e instanceof TreeNode)
                         ((TreeNode<K,V>)e).split(this, newTab, j, oldCap);
                     else { // preserve order
                         //低位链表的头结点、尾节点
                         Node<K,V> loHead = null, loTail = null;
                         //高位链表的头节点、尾节点
                         Node<K,V> hiHead = null, hiTail = null;
                         Node<K,V> next;
                         do {
                             //链表读取
                             next = e.next;
                             //这里又是一个利用位运算 代替常规运算的高效点：
                             // 利用哈希值 与 旧的容量，可以得到哈希值去模后，是大于等于oldCap还是小于oldCap，
                             // 等于0代表小于oldCap，应该存放在低位，否则存放在高位
                             if ((e.hash & oldCap) == 0) {
                                 if (loTail == null)
                                     loHead = e;
                                 else
                                     loTail.next = e;
                                 loTail = e;
                             }
                             else {
                                 if (hiTail == null)
                                     hiHead = e;
                                 else
                                     hiTail.next = e;
                                 hiTail = e;
                             }
                         } while ((e = next) != null);
                         if (loTail != null) {
                             loTail.next = null;
                             newTab[j] = loHead;
                         }
                         if (hiTail != null) {
                             hiTail.next = null;
                             newTab[j + oldCap] = hiHead;
                         }
                     }
                 }
             }
         }
         return newTab;
     }

 

HashMap、Hashtable

  1. HashMap和Hashtable的区别。

    通过上面的put等方法可知HashMap并不是线程安全的，而Hashtable就是HashMap的线程安全版本。Hashtable内部的实现还是跟HashMap不太一样。

  • HashMap中key和value均可以为null，但是Hashtable中key和value均不能为null。
  • HashMap在JDK1.8采用的是数组+链表/红黑树结构实现，而Hashtable中采用的是数组+链表实现。
  • HashMap中出现hash冲突时，如果链表节点数小于8时是将新元素加入到链表的末尾，而Hashtable中出现hash冲突时采用的是将新元素加入到链表的开头。
  • HashMap中的hash算法是位运算按位与，Hashtable中是采用的是求余数。
  • HashMap中默认容量的大小是 16 ，数组容量的大小要求是 2的n次，而计算hashcode时进行的是位运算，因为模数是 2 的幂，可以直接使用位运算来得到结果，效率要大大高于求余运算，并且为了减少哈希冲突将 高16位与低 16位进行亦或。Hashtable中默认数组容量是 11 ，扩容扩大到2n+1，当哈希表的大小为素数时，简单的取模哈希的结果会更加均匀。
  • HashMap不是线程安全的，而Hashtable是线程安全的，因为Hashtable中的get和put方法均采用了synchronized关键字进行了方法同步。但是由于同步锁的使用，Hashtable的效率往往比HashMap低。
  • HashMap采用的是Iterator迭代器，可以在遍历时移除元素，Hashtable使用的是Enumeration迭代器，只读，不能移除元素。
