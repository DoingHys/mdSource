2、synchronized的实现

加锁原理

当synchronized 同步代码块的时候，会在代码块的起始位置插入 monitorenter指令，在结束或异常处插入 monitorexit指令。当执行到monitorenter 指令时，将会尝试获取对象所对应的 monitor 的所有权，即尝试获得对象的锁，而 synchronized用的锁是存放在 Java对象头 中的。

在 Java 中synchronized关键字是能在操作同步资源时直接加锁，实现线程同步的锁，并且实现的是悲观锁。悲观锁会使同步资源在同一时间只有一个线程可以访问，大大的增加资源的安全性的同时，牺牲掉了系统性能。为了在一定程度上减少获得锁和释放锁带来的性能消耗，在 JDK1.6 之后便引入了偏向锁和轻量级锁。

在synchronized中锁的级别由低到高依次为：无锁状态、偏向锁状态、轻量级锁状态、重量级锁状态。这几个状态会随着竞争情况逐渐升级。

            |--------------------------------------------------------------------------------|
            |   unused:25 | identity_hashcode:31 | unused:1 | age:4 | biased_lock:1 | lock:2 |  无锁状态
            |--------------------------------------------------------------------------------|
            |   thread:54 |         epoch:2      | unused:1 | age:4 | biased_lock:1 | lock:2 |  偏向锁状态
            |--------------------------------------------------------------|--------|--------|
            |                     ptr_to_lock_record:62                             | lock:2 |  轻量锁状态
            |--------------------------------------------------------------|--------|--------|
            |                     ptr_to_heavyweight_monitor:62                     | lock:2 |  重量锁装填
            |--------------------------------------------------------------|--------|--------|
            |                                                                       | lock:2 |  GC状态
            |--------------------------------------------------------------------------------|

lock:2：代表锁状态标记位。

biased_lock:1：是否启用偏向锁标记位。当该位为1时表示对象启用了偏向锁。

这两位组合起来代表着锁的状态：

age:4：代表对象的GC年龄。

identity_hashcode：对象的hashcode。

epoch：偏向时间戳。

thread：持有偏向锁的线程。

ptr_to_lock_record：指向栈中锁记录的指针。

ptr_to_heavyweight_monitor：指向管程Monitor的指针

由于CPU有两种不同的字节序，所以需要根据当前CPU的字节序来判断上述Mark Word的内容。

使用System.out.println(ByteOrder.nativeOrder());可以查看当前CPU的字节序。输出LITTLE_ENDIAN则是小端序。本次实验也是小端序。

  • 小端序：数据的高位字节存放在地址的高端 低位字节存放在地址低端（如 0x1234567，显示的是67 45 23 01）
  • 大端序：数据的高位字节存放在地址的低端 低位字节存放在地址高端（如 0x1234567，显示的是01 23 45 67，符合阅读习惯）

不同锁状态下的对象头

1、无锁状态

代码如下：

xxxxxxxxxx
 SyncLock syncLock = new SyncLock();  //新建对象
 System.out.println(ClassLayout.parseInstance(syncLock).toPrintable());   //打印对象

打印内容：

此时Mark Word是00000000 00000000 00000000 00000000 00000000 00000000 00000000 00000001，根据上面表格，无锁时，锁标记位为01，偏向锁为0，因为是在main方法中执行，不存在GC，所以age也为0

unused:1 | age:4 | biased_lock:1 | lock:2

0 0000 0 01

无锁状态的特点是没有对资源进行锁定，所有线程都能访问并修改同一个资源，但同时只有一个线程能修改成功。修改操作会在循环内进行，线程会不断的尝试修改共享资源。如果没有冲突就修改成功并退出，否则就会继续循环尝试（自旋）。如果有多个线程修改同一个值，必定会有一个线程能修改成功，而其他修改失败的线程会不断重试直到修改成功。

2、偏向锁状态

代码如下：

新建一个线程池，线程内循环的执行锁操作。这里为什么要sleep 5s是因为JVM启动时会进行一系列的复杂活动，比如装载配置，系统类初始化等等。在这个过程中会使用大量synchronized关键字对对象加锁，JVM默认延时加载偏向锁，如果不加这个sleep会导致偏向锁膨胀，变成轻量级锁。

xxxxxxxxxx
 Thread.sleep(5000L);
 ExecutorService executorService = Executors.newFixedThreadPool(1);
 SyncLock syncLock = new SyncLock();
 executorService.execute(() -> {
   for (int j = 0; j < 10; j++) {
      synchronized (syncLock) {
          System.out.println(ClassLayout.parseInstance(syncLock).toPrintable());
      }
   }
});

打印内容：

此时Mark Word是00000000 00000000 01111111 11010000 00110101 10100111 00011000 00000101，根据上面表格，锁标记位为01，偏向锁为1。

unused:1 | age:4 | biased_lock:1 | lock:2

0 0000 1 01

偏向锁是当在一段同步代码一直被同一个线程所访问，不存在多个线程竞争，那个该线程自动获得锁，从而降低了获取锁带来的消耗，提高了锁的性能。当一个线程获得偏向锁时，会在Mark Word (上面表格里的thread)上记录该线程的线程ID。当该线程再次进入的时候，直接检测线程ID是否与Mark Word中的相同，从而减少了加锁、解锁带来的CAS操作，替换锁时也只需要一次CAS操作即可。

关于偏向锁的撤销，需要等待全局安全点，即在某个时间点上没有字节码正在执行时，它会先暂停拥有偏向锁的线程，然后判断锁对象是否处于被锁定状态。如果线程不处于活动状态，则将对象头设置成无锁状态，并撤销偏向锁，恢复到无锁（标志位为01）或轻量级锁（标志位为00）的状态。

3、轻量级锁

代码如下：

xxxxxxxxxx
 public static void main(String[] args) throws InterruptedException {
        Thread.sleep(5000L);
        ExecutorService executorService = Executors.newFixedThreadPool(2);
        SyncLock syncLock = new SyncLock();
        executorService.execute(() -> {
            for (int j = 0; j < 4; j++) {
                synchronized (syncLock) {
                    System.out.println(ClassLayout.parseInstance(syncLock).toPrintable());
                }
                // 最后一次时继续保持线程，防止偏向锁被撤销
                if(j == 3){
                    try {
                        Thread.sleep(5000L);
                    } catch (InterruptedException ignore) {
                    }
                }
            }
        });
        // 等待上面的线程获得偏向锁
        Thread.sleep(4000L);
        executorService.execute(() -> {
            synchronized (syncLock) {
                System.out.println(ClassLayout.parseInstance(syncLock).toPrintable());
            }
        });
    }

打印内容：

图上半部分是偏向锁。

图下半部分，Mark Word是00000000 00000000 01110000 00000000 00000110 10011011 11000111 01110000 锁标记为是 00 是轻量级锁。

ptr_to_lock_record:62 | lock:2

00000000 00000000 01110000 00000000 00000110 10011011 11000111 011100 00

轻量级锁是当锁为偏向锁时，有多个线程进行竞争，此时偏向锁就会升级为轻量级锁，其他线程会通过自旋来尝试获取锁，此时线程不会阻塞，减少了线程切换，提高了性能。在代码进入同步块的时候，如果同步对象锁状态为无锁状态，虚拟机将首先在当前线程的栈帧中建立一个名为锁记录Lock Record的空间，用于存储锁对象目前的 Mark Word 的拷贝，然后将对象头中的 Mark Word复制到锁记录中。

5、重量级锁

代码如下：

xxxxxxxxxx
    public static void main(String[] args) throws InterruptedException {
        Thread.sleep(5000L);
        SyncLock syncLock = new SyncLock();
        ExecutorService executorService = Executors.newFixedThreadPool(2);
        for (int i = 0; i < 2; i++) {
            executorService.execute(() -> {
                synchronized (syncLock) {
                    System.out.println(ClassLayout.parseInstance(syncLock).toPrintable());
                }
            });
        }
    }

打印内容：

此时Mark Word是00000000 00000000 01111111 10010111 01011110 10000000 00110110 00001010，根据上面表格，锁标记位为10，是重量级锁。

ptr_to_heavyweight_monitor:62 | lock:2

00000000 00000000 01111111 10010111 01011110 10000000 00110110 000010 10

在轻量级锁时，当前只有一个线程自旋进行等待，自旋超过一定的次数时，轻量级锁便会升级为重量级锁。如果当一个线程已持有轻量级锁，另一个线程在自旋时，又有第三个线程竞争，轻量级锁也会升级为重量级锁。当有一个线程获取重量级锁之后，其余所有等待获取该锁的线程都会处于阻塞状态。

重量级锁通过对象内部的监视器monitor实现，而其中 monitor 的本质是依赖于底层操作系统的 Mutex Lock 实现，操作系统实现线程之间的切换需要从用户态切换到内核态，切换成本非常高，如果出现频繁地对线程运行状态的切换，线程的挂起和唤醒，就会消耗大量的系统资源，导致性能低下。

3、CAS

CAS 全称是 compare and swap，即比较并交换，更新值时先比较当前值是否与读取出来时一样，如果一样就更新，如果不一样就重试（自旋）更新，它是一种原子操作。CAS底层使用JNI调用C代码实现的，这里就不展开了。

当然CAS也不是百分百可靠的，它也会带来问题。

  • ABA问题，CAS在操作值的时候会检查值是否发生变化，如果没有发生变化则更新；如果一个值原来是A，变成了B，又变成了A，那么使用CAS进行检查时会发现它的值没有发生变化，但是实际上却变化了。 解决这一问题的思路是使用版本号，在变量前面追加上版本号，每次变量更新的时候把版本号加一，那么A-B-A 就会变成1A-2B-3A。 目前在JDK的atomic包里提供了一个类AtomicStampedReference来解决ABA问题。这个类的compareAndSet方法作用是首先检查当前引用是否等于预期引用，并且当前标志是否等于预期标志，如果全部相等，则以原子方式将该引用和该标志的值设置为给定的更新值。
  • 长时间自旋问题，如果CAS操作不成功，则会原地自旋，如果长时间自旋会给CPU带来非常大的执行开销。
  • 只能保证一个共享变量的原子操作，当对一个共享变量执行操作时，我们可以使用循环CAS的方式来保证原子操作，但是对多个共享变量操作时，循环CAS就无法保证操作的原子性，这个时候就可以用锁来保证原子性。
